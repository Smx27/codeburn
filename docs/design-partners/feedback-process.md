# Feedback Process

How to provide feedback as a design partner.

## Overview

Your feedback helps us build a better product. This document describes how to provide effective feedback.

## Types of Feedback

### Bug Reports

Something isn't working as expected.

**When to report:**
- Feature doesn't work
- Error occurs
- Unexpected behavior
- Performance issue

**How to report:**
- Use [Bug Report Template](bug-report-template.md)
- Include steps to reproduce
- Include screenshots
- Include environment info

### Feature Requests

You want something new or different.

**When to request:**
- Missing functionality
- Better workflow
- New capability
- Improvement to existing feature

**How to request:**
- Create GitHub issue with label `feature-request`
- Describe the problem you're solving
- Describe your ideal solution
- Explain why it's important

### General Feedback

Your overall experience and impressions.

**When to share:**
- Something works well
- Something is confusing
- Something is frustrating
- You have suggestions

**How to share:**
- Post in Slack #aiinsight-design-partners
- Email design-partners@aiinsight.dev
- Share during weekly check-in

## Feedback Channels

### Slack (Preferred)

**Channel:** #aiinsight-design-partners

**Best for:**
- Quick questions
- Bug reports
- Feature discussions
- General feedback

**Tips:**
- Use threads for discussions
- Tag @team for urgent issues
- Share screenshots when possible
- Be specific about what you're seeing

### Email

**Address:** design-partners@aiinsight.dev

**Best for:**
- Detailed feedback
- Sensitive issues
- Off-topic discussions
- Formal requests

**Tips:**
- Use descriptive subject lines
- Include relevant context
- Attach screenshots if helpful
- Allow 24-hour response time

### GitHub Issues

**Repository:** https://github.com/getagentseal/codeburn/issues

**Best for:**
- Bug reports
- Feature requests
- Documentation issues
- Technical discussions

**Tips:**
- Use issue templates
- Label appropriately
- Include reproduction steps
- Link to related issues

### Weekly Check-ins

**When:** Every Thursday at 10 AM PT

**Best for:**
- Demoing features
- Discussing roadmap
- Deep feedback sessions
- Relationship building

**Tips:**
- Prepare feedback in advance
- Prioritize your topics
- Take notes
- Follow up on action items

## Feedback Quality

### Good Feedback

```
**What I was trying to do:**
View my usage for the past week.

**What I expected:**
See a breakdown by provider.

**What actually happened:**
The providers page showed "No data available" even though I have data.

**Environment:**
- OS: macOS 14.5
- AIInsight: v1.0.0
- Browser: Chrome 125

**Screenshots:**
[Attached]

**Suggested fix:**
Maybe check if data exists before showing "No data available"?
```

### Bad Feedback

```
It doesn't work.
```

### What Makes Good Feedback

1. **Specific** — Exactly what happened
2. **Reproducible** — Steps to see it yourself
3. **Contextual** — Environment and setup
4. **Actionable** — Clear what should happen
5. **Polite** — Constructive tone

## Feedback Lifecycle

### 1. Submission

You submit feedback via one of the channels.

### 2. Acknowledgment

We acknowledge receipt within 24 hours.

### 3. Triage

We categorize and prioritize the feedback.

### 4. Action

We fix bugs, implement features, or address concerns.

### 5. Follow-up

We let you know what we did and why.

## Priority Levels

| Priority | Description | Response Time |
|----------|-------------|---------------|
| Critical | System down, data loss | 4 hours |
| High | Major feature broken | 24 hours |
| Medium | Minor feature issue | 1 week |
| Low | Enhancement, nice-to-have | 2 weeks |

- **Email** — Formal communications

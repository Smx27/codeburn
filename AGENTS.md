# AGENTS.md

## Monorepo layout

npm workspaces. Root builds a CLI (`src/main.ts` → `dist/cli.js` via tsup).

```
packages/
  analytics-engine/   — aggregation queries over Postgres
  auth-shared/        — JWT helpers, isApiKey(), shared auth logic
  distribution/       — npm publish packaging
  platform/           — platform detection, file paths
  sync-engine/        — agent log sync to cloud

apps/
  ingestion-api/      — Express, port 3001. Receives agent events, stores in Postgres.
  dashboard-api/      — Express, port 3002. Dashboard queries, auth, email.
  dashboard-web/      — Next.js, port 3000. Frontend SPA.
```

Internal packages are referenced as `@aiinsight/*` (wildcard deps in workspace package.json files).

## Build & dev commands

```bash
# Root CLI
npm run build              # tsup bundle + copy cli.ts
npm run dev                # tsx src/cli.ts

# Per-workspace
npm run api:build          # ingestion-api tsc
npm run api:dev            # ingestion-api nodemon
npm run dashboard-api:build
npm run dashboard-api:dev
npm run dashboard-web:build
npm run dashboard-web:dev

# Shared packages (must build before APIs that depend on them)
npm run analytics:build    # packages/analytics-engine
npm run sync:build         # packages/sync-engine
```

**Build order matters**: `auth-shared` → `analytics-engine` → apps. The Dockerfiles handle this automatically. Locally, run shared builds first if you change them.

## Testing

```bash
npm test                   # vitest (root tests/ directory only)
npx vitest run tests/parser-filter.test.ts   # single file
```

~90 test files in `tests/`, mostly covering the root CLI parser/provider logic. No test suites exist inside `apps/` or `packages/` — those are tested via the root vitest config.

## TypeScript

All packages use ESM (`"type": "module"`) with `"moduleResolution": "bundler"`. Strict mode enabled everywhere.

- Root: `tsconfig.json` — `rootDir: src`, `outDir: dist`
- ingestion-api and dashboard-api: same pattern, plus `@/*` path alias → `src/*`
- dashboard-web: Next.js config, `@/*` → `./src/*`, noEmit (Next handles compilation)

## Docker

`docker-compose.yml` at root. PostgreSQL on host port 5433 (internal 5432).

**Key**: use `expose` not `ports` for services — Coolify manages ingress.

**Entrypoint**: `ingestion-api` runs migrations automatically on start via `docker-entrypoint.sh`.

**Seed commands** (run inside container):
```bash
docker compose exec ingestion-api node dist/seed/seed.js --size small
docker compose exec ingestion-api node dist/seed/demo.js
```

**Build args for dashboard-web**: `NEXT_PUBLIC_API_URL` and `NEXT_PUBLIC_APP_URL` must be set at build time (validated by `next.config.js`). In Coolify, set these to external URLs.

**Environment** (all required via `.env`):
- `POSTGRES_PASSWORD` — fail-fast if missing
- `JWT_SECRET` — shared between ingestion-api and dashboard-api
- See `.env.docker.example` for full template

## Seed scripts

Seed data is generated by TypeScript factories in `apps/ingestion-api/src/seed/factories/`. The compiled JS lives at `dist/seed/`.

**Gotcha**: node-postgres cannot infer parameter types in large multi-row INSERT statements with `$N` placeholders. The demo seed (`demo.ts`) uses inline SQL values via a `sqlVal()` helper instead of parameterized queries. If you add new aggregation inserts, use the same pattern or cast every parameter explicitly (e.g. `$1::uuid`).

## CI

Only a release workflow (`.github/workflows/release.yml`) — builds SEA binaries on tag push. No lint/test CI pipeline.

## Key architectural notes

- The root CLI parses local agent session files (Claude, Codex, Cursor, etc.) and renders TUI dashboards
- Apps form a cloud deployment: ingestion-api receives agent telemetry, dashboard-api serves analytics, dashboard-web renders the UI
- `auth-shared` is the shared JWT/auth library used by both APIs
- `analytics-engine` contains SQL aggregation queries used by dashboard-api
- Migrations live in `apps/ingestion-api/src/database/migrations/` and run sequentially on ingestion-api startup
